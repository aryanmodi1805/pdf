All SRS v1.0 functional requirements — FR-ADM-001 through FR-CLI-006, FR-SEL-001 through FR-SEL-007 (seller-side listing management, pricing & escalation), FR-ADM-FIN-001 through FR-ADM-FIN-008 (admin financial dashboard & reports), FR-ADM-006 (seller blocking cascade) — have been implemented and verified across both the backend and Flutter app.

Architectural Overview

The core decision was a strict separation between Admin-controlled Templates and Seller-specific Listings — as mandated by SRS 4.1.

BookingServiceTemplate

Created & managed by Admin only. Defines the service category, price range limits, visibility, and default listing status. Acts as the blueprint for all seller listings.

BookingListing

Created by Sellers from a template. seller_price is set by seller; auto-calculates 15% JAD displayed price. Subject to admin moderation (Approve/Hide).

Pricing Business Rule (BR-001)

Seller Price + 15% JAD Fee = Displayed Price
At checkout (SRS 3.3.008): the App payment screen adds 15% VAT + 1 SAR processing fee on top of the Displayed Price shown in the listing

  • ▸New: Finance Dashboard NEW

  • ▸SRS FR-ADM-007: The re-assignment modal now requires "Confirm Transfer with Client" before allowing re-assignment. Unauthorized re-assignments are prevented.
  • ▸Updated: Seller Blocking Cascade CHANGED
  • ▸SRS FR-ADM-006 — Blocking a seller now instantly hides all their active Booking Listings across the platform in real time.
🏪 Seller · Mobile & Web App

SELLER

  • ▸New: Browse Service Templates NEW
  • ▸Sellers can now browse templates via the new API. Each template shows price range, category, and description, giving sellers a clear catalog of services they can offer.
  • ▸New: Create Booking Listings NEW
  • ▸Sellers set their own price based on template constraints. Backend auto-calculates 15% JAD fee and displayed price. New listing status is "Pending Approval" until admin approves.
  • ▸New: Auto Wallet Payout NEW
  • ▸On order confirmation, the seller's wallet is automatically credited with seller_base_price. No manual request needed — enforced via the OrderObserver.
  • ▸New: 2-Hour Response Times with Quiet Period NEW
  • ▸Sellers have 2 hours to respond to Booking Orders. The timer is paused during 7:00 PM – 9:00 PM (Quiet Period) per SRS 3.7. Failure to respond escalates the order automatically.
  • ▸New: 8-Hour Repeat Push Notifications NEW
  • ▸For unconfirmed orders, sellers receive repeating push notification reminders every 8 hours via a scheduled background task, ensuring no order is missed.
👤 Customer / User · Mobile App

CUSTOMER

  • ▸New: Booking vs Quote Differentiation NEW
  • ▸The services listing now clearly shows each service's type — "booking" = fixed price / "quote" = negotiation. Backend returns type + lowest available price from active seller listings.
  • ▸New: Direct Checkout — No quote required NEW
  • ▸Customers can book and pay instantly. The checkout screen shows a full breakdown: Service price + 15% VAT + 1 SAR fee. Order is created immediately in "Paid" status.
  • ▸Improved: Fully Transparent Pricing CHANGED
  • ▸SRS FR-CLI-002 — Customers always see the full price breakdown before confirming. No hidden fees. VAT and processing fee shown as separate line items at checkout.
  • ▸New: 2-Hour Escalation Protection NEW
  • ▸SRS FR-CLI-003 — If a seller fails to confirm within 2 hours, the order is automatically escalated to admin for re-assignment. Customer is notified at each status change.

New API Endpoints (Backend)

Method Endpoint Auth Purpose SRS Ref
POST /api/customer/booking/checkout Customer Token Direct checkout — validates listing, charges wallet, creates Order in "Paid" status, dispatches EscalateOrderJob FR-CLI-001
GET /api/seller/booking/templates Seller Token Returns all visible BookingServiceTemplates admin has published; includes price range, category, description FR-SEL-001
POST /api/seller/booking/listing Seller Token Seller creates a listing from a template. Accepts template_id + seller_price. Auto-computes jad_fee + displayed_price. Status: "Pending" FR-SEL-002
GET /api/seller/booking/my-listings Seller Token Returns all listings created by the authenticated seller with template and category info FR-SEL-003
PUT /api/seller/booking/listing/{id} Seller Token Update listing price (re-validated against template min/max), description, or images. JAD fee recalculated FR-SEL-003
POST /api/seller/booking/listing/{id}/publish Seller Token Publishes a listing. Status follows template's default_listing_status (Published or Pending) FR-SEL-006
GET /api/seller/booking/orders Seller Token Seller order inbox: all incoming orders with client, listing, and template details FR-SEL-007
POST /api/seller/booking/orders/{id}/accept Seller Token Accept order → status "Confirmed", triggers wallet payout via OrderObserver FR-SEL-007
POST /api/seller/booking/orders/{id}/reject Seller Token Reject with required reason (Busy / Product Not Available) + optional note. Order → Cancelled, Payment → Refunded FR-SEL-007

Backend Auth Fix — BookingController CHANGED

The BookingController was missing Sanctum authentication, causing all seller booking API calls to return HTTP 500.

Problem

• auth()->id() used in 9 places — returns null without Sanctum middleware
• Seller routes have no explicit auth:sanctum middleware group
• Error: Attempt to read property "id" on null

Solution

• Added $this->middleware('auth:sanctum') to constructor (matching SellerProfileController pattern)
• Replaced all auth()->id() → $request->user()->id
• Added Request $request parameter to 5 methods that were missing it

After

• Failure: AlertDialog with red ❌ icon — "Failed to create listing. Please try again."
